# 1주차 연구노트

## 진행 목표

1주차의 목표는 전체 프로젝트의 연구 주제와 범위를 확정하고, 이후 개발을 진행할 수 있는 기본 환경을 준비하는 것이었다. 연구 주제는 컨테이너 환경에서 동작할 수 있는 고가용성 소프트웨어 로드밸런서를 개발하는 방향으로 정리하였다. 이번 주차에서는 실제 기능 구현보다 프로젝트가 어떤 문제를 다루고, 어떤 순서로 구현과 검증을 진행할지 정하는 데 중점을 두었다.

또한 Go 기반으로 reverse proxy와 로드밸런싱 기능을 구현할 수 있도록 개발 환경을 구성하고, 레포지토리 구조와 작업 컨벤션을 정리하는 것을 목표로 하였다. 이후 구현과 테스트를 일관되게 진행하려면 설정 파일, 코드 구조, 문서 관리 방식이 먼저 정리되어 있어야 하므로, 1주차에는 프로젝트 기반을 마련하는 작업을 우선하였다.

## 진행 내용

먼저 연구 주제와 전체 범위를 정리하였다. 프로젝트의 핵심 주제는 단순한 HTTP reverse proxy 구현이 아니라, 컨테이너 환경에서 사용할 수 있는 소프트웨어 로드밸런서를 직접 구현하고 고가용성 구조까지 확장하는 것이다. 따라서 1~8주차는 로드밸런싱 PoC 구현과 성능 비교를 중심으로 잡고, 9~16주차는 클러스터링 기능 구현과 장애조치 검증으로 이어지는 구조로 정리하였다. 이렇게 범위를 나눈 이유는 로드밸런싱 기능과 고가용성 기능을 한 번에 구현하기보다, 먼저 단일 노드에서 요청 처리와 backend 선택 구조를 검증한 뒤 클러스터링 기능으로 확장하는 편이 전체 흐름을 관리하기 쉽기 때문이다.

기능 요구사항도 최종 프로젝트에서 보여줄 기능을 기준으로 큰 단위에서 정리하였다. 첫 번째 요구사항은 클라이언트 요청을 받아 적절한 backend 서버로 전달하는 reverse proxy 기능이다. 두 번째 요구사항은 여러 backend 서버 사이에 요청을 분산하는 로드밸런싱 기능이며, Round Robin, Hash, Least Connections 같은 알고리즘을 비교할 수 있는 구조를 목표로 하였다. 세 번째 요구사항은 backend 상태를 확인하고 장애가 있는 서버를 요청 대상에서 제외하는 health check 기능이다. 네 번째 요구사항은 여러 로드밸런서 노드가 함께 동작할 수 있는 고가용성 기능이며, 장애 상황에서도 서비스 진입점이 유지되도록 리더 선출과 VIP 점유 기능을 포함하는 방향으로 정리하였다. 마지막으로 구현 결과를 기존 오픈소스 환경과 비교할 수 있도록 성능 테스트와 장애조치 테스트를 함께 수행하는 것을 요구사항에 포함하였다.

개발 언어는 Go를 기준으로 잡았다. Go는 표준 라이브러리에서 HTTP 서버와 클라이언트 기능을 제공하고, `net/http/httputil`을 통해 reverse proxy 구현의 기본 구조를 확인할 수 있다. 또한 단일 바이너리로 실행하기 쉽고, Docker 기반 환경에서 테스트하기에도 적합하다. 이번 프로젝트에서 다룰 요청 처리, 상태 관리, health check는 동시성이 필요한 작업을 포함하므로 Go의 동시성 모델도 프로젝트 목적에 맞는다고 판단하였다.

레포지토리 구조는 구현 코드, 설정 파일, 문서, 작업 계획을 분리하는 방향으로 정리하였다. 실제 구현 코드는 `internal/` 아래에 두고, 실행 및 프록시 설정은 `configs/`, 아키텍처와 API 문서는 `docs/`, 작업 계획과 연구노트 초안은 `plan/`에 보관하는 방식으로 잡았다. 이 구조를 사용하면 구현 코드와 보고서 작성 자료가 섞이지 않고, 각 파일의 목적을 기준으로 자료를 찾을 수 있다. 특히 `plan/`은 연구노트, 작업 계획, 벤치마크 결과처럼 작업 중 참고할 자료를 보관하는 영역으로 활용하기로 하였다.

프로젝트 컨벤션도 함께 정리하였다. `main.go`는 실행 진입점으로 얇게 유지하고, 실제 정책과 비즈니스 로직은 `internal/` 아래에 역할별로 나누어 두는 방향을 잡았다. 설정 로드, 요청 경로 해석, backend 선택, 상태 관리, 실제 요청 전달처럼 성격이 다른 작업은 가능한 한 분리해서 구현하기로 하였다. 이 기준은 단순히 디렉토리를 나누기 위한 것이 아니라, 이후 기능을 추가할 때 어느 영역을 수정해야 하는지 명확히 하기 위한 것이다.

## 검토 및 결과

1주차 작업을 통해 프로젝트의 전체 방향을 로드밸런싱 PoC와 고가용성 클러스터링 확장으로 나누어 정리할 수 있었다. 1~8주차에는 reverse proxy 기반의 로드밸런싱 엔진을 구현하고, 성능 비교와 리팩토링을 통해 구조를 개선한다. 9~16주차에는 클러스터링 기능과 장애조치 검증을 다루면서 단일 노드 구조를 고가용성 구조로 확장한다. 이 흐름을 정리한 덕분에 각 주차의 작업 목적이 서로 분리되지 않고 하나의 연구 과정으로 이어지게 되었다.

개발 환경 측면에서는 Go 프로젝트를 기준으로 레포지토리 구조와 문서 관리 방식을 정했다. `configs/`, `internal/`, `docs/`, `plan/`, `scripts/`, `tools/` 같은 주요 디렉토리의 역할을 구분함으로써 개발과 문서 작성, 테스트 보조 자료를 별도로 관리할 수 있게 되었다. 특히 이후 주차에서 성능 테스트 결과나 설계 변경 사항이 계속 생길 수 있으므로, 문서와 계획 자료의 위치를 정한 것은 연구 진행 과정 기록에 중요하다.

## 다음 주차 계획

2주차에는 로드밸런싱 PoC를 진행하기 전에 기존 오픈소스 로드밸런서를 조사할 계획이다. 특히 Nginx와 HAProxy의 내부 구조, 설정 방식, upstream/backend 관리 방식, 로드밸런싱 정책을 확인하고 자체 구현에서 참고할 기준을 정리한다.

또한 기능 요구사항과 유저플로우를 더 구체화하고, Go에서 reverse proxy를 구현하기 위해 필요한 기술과 라이브러리를 조사한다. 이 조사 결과는 3주차의 실제 reverse proxy 기능 구현과 관리자 대시보드 1차 구현의 기준으로 사용할 예정이다.
